SS — Safe Subset Language
A Proof-by-Construction Compiler for Automated Kernel Verification
================================================================================
Author:  Yumin
Date:    2026-07-09
Origin:  Derived from Y language safe block system (YSU Engine)

1. MOTIVATION
-------------

Current formal verification workflows (CBMC, Z3, herd7, Coq) all share the
same fundamental problem: the proof is EXTERNAL to the code.

You write code, then you separately write proofs about it. Two independent
artifacts exist. Humans make mistakes in both steps. The proofs can go
stale as the code changes. The toolchain is fragile (missing Z3 binary,
broken AUR packages, unsound pointer concurrency in CBMC, etc.).

The CachyOS kernel verification project exposed all of these pain points:
  - CBMC's concurrency engine throws "pointer handling for concurrency is
    unsound" for any threaded code
  - herd7 LKMM litmus tests require exact parser syntax that is non-obvious
  - Harnesses must be manually written and kept in sync with kernel patches
  - The verification chain depends on external tools being installed and
    compatible

SS solves this by making the language itself the proof system.


2. CORE PRINCIPLE
-----------------

  "If an SS program compiles, it is formally proven correct."

There is no separate verification step. Compilation IS verification.

This is achieved by a single, absolute rule:

  SS IS CONSTRUCTED ENTIRELY FROM Y LANGUAGE "safe" BLOCKS.
  NOTHING ELSE EXISTS IN SS.

Y's "safe" blocks are formally verified primitives. Each one has a machine-
checked proof that it cannot:
  - Overflow or underflow any integer operation
  - Produce a division-by-zero
  - Access memory out of bounds
  - Introduce a data race in concurrent contexts
  - Enter a deadlock or livelock

Because SS programs can ONLY be composed from these building blocks, any
valid SS program inherits all of these guarantees automatically. The
programmer does not write proofs. The compiler does not run a separate
solver. The type structure of the program IS the proof.


3. COMPARISON WITH EXISTING TOOLS
----------------------------------

Tool         | Approach                          | Limitation
-------------|-----------------------------------|---------------------------
CBMC         | Bounded model checking of C       | External, manual harnesses,
             |                                   | unsound concurrency
Z3           | SMT solving                       | External, query formulation
             |                                   | requires expertise
herd7 / LKMM | Memory model litmus tests         | External, fragile syntax,
             |                                   | not automated
Coq          | Interactive theorem proving       | Manual proof tactics,
             |                                   | expert-only
Dafny        | Verification annotations in code  | Human writes proof hints
F* / Lean    | Dependent types with SMT backend  | Complex type system,
             |                                   | high learning curve
Rust         | Memory safety by ownership types  | Math correctness not covered
SS           | Proof by construction — only      | Expressiveness limited to
             | safe blocks can compose           | available safe block library
             |                                   | (this is a feature, not a bug)


4. HOW SS DIFFERS FROM COQ
---------------------------

Coq is a proof ASSISTANT. You manually write every proof step using tactics:

  Theorem no_overflow : forall penalty scale,
    penalty * scale <= MAX_BURST_PENALTY.
  Proof.
    intros. apply bound_lemma. exact H.
  Qed.

Every line of that proof is written by a human expert. Coq checks it.
This requires deep expertise and does not scale to automated pipelines.

SS is a proof-by-CONSTRUCTION compiler. The proof is implicit:

  let penalty = safe_burst_penalty(burst_time)

If safe_burst_penalty is a Y safe block, the compiler has already proven it.
No tactics. No proof steps. No human intervention.

The difference is:
  Coq  = Human writes proof, machine checks it
  SS   = Machine derives proof from type structure, human writes nothing


5. THE AUTOMATED PIPELINE
--------------------------

SS is designed from the start for a fully automated batch verification
pipeline:

  SS File 1 -> SS Compiler -> Proof derived from safe block types -> PASS -> next
  SS File 2 -> SS Compiler -> Proof derived from safe block types -> PASS -> next
  SS File 3 -> SS Compiler -> safe_x not composable with safe_y   -> FAIL -> report
  SS File N -> SS Compiler -> Proof derived from safe block types -> PASS -> done

No human in the loop. No running CBMC. No writing harnesses.
The compiler is the verifier, the verifier is the compiler.

This mirrors what the Lean 4 mathlib project does for mathematics —
automated batch verification of hundreds of theorems. SS applies the same
concept to systems code.

For CachyOS kernel verification:
  Before SS:
    Kernel patch lands -> Developer manually writes C harness ->
    Runs CBMC with Z3 -> Checks output -> Updates harness if patch changes

  With SS:
    Kernel patch lands -> SS compiler runs on SS model of subsystem ->
    PASS or FAIL reported automatically


6. THE SAFE BLOCK LIBRARY (KERNEL DOMAIN)
------------------------------------------

SS for kernel scheduler verification needs approximately 20-30 Y safe blocks.
Based on the CachyOS verification work, these are the required domains:

  6.1 Safe Arithmetic
      - safe_log2_fixed_point(v: u64, fp: u8) -> u32
        Proven: no undefined shift behavior, returns 0 for v=0
      - safe_burst_penalty(burst_time: u64) -> u32
        Proven: result always <= MAX_BURST_PENALTY
      - safe_binary_smooth(new_val: u32, old_val: u32) -> u32
        Proven: result always in [min(new,old), max(new,old)]
      - safe_scale_weight(value: u64, weight: u64) -> u64
        Proven: no overflow for realistic kernel weight ranges
      - safe_div_nonzero(a: u64, b: u64) -> u64
        Proven: b is guaranteed nonzero by type, no division-by-zero

  6.2 Safe Virtual Time
      - safe_vruntime_cmp(a: u64, b: u64) -> Ordering
        Proven: signed subtraction comparison is transitive
        under bounded wrap-around (< 2^63 difference)
      - safe_vruntime_sub(a: u64, b: u64) -> s64
        Proven: result interpreted as signed, no logic error

  6.3 Safe Concurrent Memory Access
      - safe_atomic_read(loc: &AtomicU64) -> u64
        Proven: single-word read is atomic on all supported architectures,
        no tearing possible
      - safe_atomic_write(loc: &AtomicU64, val: u64)
        Proven: single-word write is atomic, no partial update
      - safe_rcu_read(ptr: &RcuProtected<T>) -> &T
        Proven: pointer is valid for duration of RCU read-side critical section

  6.4 Safe Memory State Transitions
      - safe_reclaim_chunk(cache: u64) -> (u64, u64)
        Proven: result (reclaimed, remaining) never exceeds input cache
      - safe_swap_chunk(anon: u64, swap_free: u64) -> u64
        Proven: swap amount never exceeds min(anon/8, swap_free)
      - safe_oom_free(anon: u64) -> (u64, u64)
        Proven: freed amount is exactly half of input

  6.5 Safe Topology
      - safe_cpu_cluster_select(mask: u64, cluster_mask: u64) -> Option<usize>
        Proven: returns None if intersection is 0, else returns valid bit index
      - safe_bitmask_pack(flags: [u8; 64]) -> u64
        Proven: packed bit i corresponds exactly to flags[i] != 0

  6.6 Safe ZRAM Sizing
      - safe_kib_to_bytes(kib: u64) -> u64
        Proven: no overflow for inputs up to 16 TiB
      - safe_mib_scale(mib: u64, factor: u64) -> u64
        Proven: no overflow for realistic RAM configurations


7. WHAT SS CANNOT EXPRESS (BY DESIGN)
--------------------------------------

The limited expressiveness of SS is not a weakness — it is the mechanism
that provides the correctness guarantee.

SS CANNOT express:
  - Raw pointer arithmetic
  - Unchecked integer arithmetic (+ without safe_ prefix)
  - Direct memory reads/writes without atomic or RCU wrappers
  - Thread spawning without a safe_thread primitive
  - Any operation not covered by the Y safe block library

If a kernel subsystem requires operations outside the safe block library,
the developer must first:
  1. Prove the new operation correct in Y (adding it to the safe block library)
  2. Then express it in SS using the new block

This creates a virtuous cycle: the safe block library grows over time,
and every new safe block that is added permanently expands what can be
automatically verified.


8. RELATIONSHIP TO Y LANGUAGE
-------------------------------

SS is not a separate language implementation. It is a CONSTRAINED DIALECT
of Y:

  Y:   Contains both safe blocks and unsafe operations
       Used for general systems programming
       Formal verification is optional

  SS:  Contains ONLY safe blocks
       Used exclusively for verification-critical code
       Formal verification is mandatory — it is the compilation process

The SS compiler is Y's compiler with unsafe operations removed from the
grammar. Any Y file that uses only safe blocks is a valid SS file.
A valid SS file is a formally verified program.

10. SUMMARY
-----------

SS is Coq for systems programmers:
  - No proof tactics to learn
  - No harnesses to write
  - No external tools to install
  - No human intervention in the verification loop

It is an automated, language-level formal verification system built on
Y's proven safe blocks, designed specifically to scale to large kernel
codebases like CachyOS.

The insight: if you can restrict what a language can express to only
pre-proven primitives, then the act of writing a program in that language
IS the act of proving it correct.
